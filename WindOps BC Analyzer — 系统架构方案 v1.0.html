1 技术栈选型及理由

1.1 完整技术栈总览

层次技术选择版本选型理由
前端框架 React 18 + TypeScript ^18.3 + ^5.5 表单密集型应用，React 受控组件模式天然适合；TypeScript 保障 14 列字段映射的类型安全。
构建工具 Vite ^6.x 冷启动 <1s，HMR 极快；CRA 已停止维护，Vite 是事实标准。
UI 组件库 Ant Design 5 ^5.22 ProTable（5,000 行分页）、Form（7 字段 + 8 成本区）、Steps（多方案对比流程感）三者都强；中文生态成熟。
状态管理 Zustand ^5.x 轻量无 boilerplate；selectors 防重复渲染天然解决 5,000 行表格性能问题。
后端框架 FastAPI (Python) ^0.115 原生 async/await，直调 Ollama API 不阻塞；自动生成 OpenAPI 文档；Pydantic v2 做 14 列字段校验。
AI 通信 httpx (async) ^0.28 FastAPI 生态首选异步 HTTP 客户端，调 localhost:11434/api/chat 无阻塞。
ORM SQLAlchemy 2.0 + Alembic ^2.0 + ^1.14 SQLite 下最成熟 Python ORM；Alembic 做 schema 迁移；2.0 async session 与 FastAPI 完美配合。
数据库 SQLite (aiosqlite) ≥3.35 零安装、单文件、零配置；5,000 行 × 14 列 ≈ 70K cells 轻松处理；WAL 模式支持读写并发。
数据 ETL Pandas ^2.2 14 列 CSV 解析、Currency/ExchangeRate 聚合计算、分组统计的自然选择。
CSV 导出 papaparse + file-saver — 前端直接拿数据构建 Blob 下载，避免后端额外端点。
包管理 Poetry + pnpm — Poetry 锁依赖确定性强；pnpm 磁盘效率高。
项目组织 Monorepo（手动 workspace） — Demo 级项目，两个顶层目录 frontend/ backend/ 各自治即可。

1.2 未采纳方案及原因

放弃方案放弃原因
Next.js / Remix纯 localhost SPA，无 SSR/SEO 需求，Next.js server component 是额外心智负担。
Streamlit / Gradio数据驱动原型快，但"工程师审核多方案对比"这类交互式 UI 弱，不可扩展为产品。
Electron浏览器已满足"本地运行"，多一层打包复杂度，且 Ollama 已是独立进程。
Django / FlaskFastAPI 的 async 原生支持对 Ollama 长耗时调用更友好；Django ORM 太重。
MongoDB / PostgreSQLSQLite 零运维，5,000 行数据规模不需要独立数据库进程。
LangChainDemo 阶段 prompt 简单明确，langchain 抽象层增加调试难度；直接调 Ollama HTTP API 即可。
Docker本地 Demo，笔记本直接 pip + npm 安装即可；Docker 增加 OCI 层复杂度。
Node.js + ExpressPython 是 AI/ML 生态母语，Pandas 处理 CSV 天然优势；Ollama 官方文档 Python 示例最丰富。

2.2 关键交互流

1. 上传 CSV：前端 FormData → POST /api/data/upload → Pandas 解析 → 写入 SQLite + 留存文件
2. AI 推断：POST /api/cases/{id}/infer → AIInferenceService 构建 Prompt → Ollama → SSE 流式返回 → 写入 case_cost_items
3. 方案生成：POST /api/cases/{id}/plans → 三次并行的 AI 调用（asyncio.gather）→ 写入 plan + plan_cost_items

4 核心模块设计

4.1 P0-1：AI 模型配置

维度说明
前端职责设置页面：Ollama 地址输入框（默认 http://localhost:11434）、模型下拉（调 /api/ai/models）、Temperature/top_p 滑块、连接测试按钮
后端职责GET /api/ai/models 调 Ollama /api/tags；配置写入 ai_config 表；连接测试调 Ollama /api/generate 空 prompt
依赖无其他模块依赖，但其他模块依赖此模块的配置

4.2 P0-2：历史数据管理 + CostGroup 映射

维度说明
前端职责拖拽上传区域（Ant Design Upload + Dragger）；上传后数据预览表格（ProTable，100 行/页）；CostGroup 映射面板：显示 6 个值 → 7 类业务成本的映射关系（可编辑）
后端职责POST /api/data/upload：Pandas 读 CSV → 校验 14 列完整性 → 批量 INSERT → COPY 到 uploads/；GET /api/data/records 分页查询；GET /api/data/summary 统计
依赖无——是数据入口
CostGroup 默认映射：Parts → 零部件成本 / Crane → 吊车成本 / Freight → 运输费用 / Labour → 人力成本 / Tools → 工具成本 / Other → 其他

4.3 P0-3：案例创建 + AI 填值

维度说明
前端职责表单分两区：🟡 用户输入（7 字段）+ 🟢 AI 推断区（8 类成本），初始置灰，点击「AI 推断」后加载动画 → 显示结果
后端职责POST /api/cases 保存元数据；POST /api/cases/{id}/infer 构建推断 prompt（含历史 CostGroup 统计摘要 + 映射表 + 用户字段）→ 调 Ollama → 解析 JSON → 写入 case_cost_items
依赖依赖 P0-2（需历史数据才有统计基础）、P0-1（AI 配置）

4.4 P0-4：工程师审核

维度说明
前端职责展示 🟢 区 8 类成本的可编辑表格；AI 推断值 vs 工程师修改值对比；🖊 已修改标记；「重置为 AI 原始值」按钮
后端职责PUT /api/cases/{id}/costs：批量 upsert，记录 is_manual_override=True
依赖P0-3

4.5 P0-5：多方案对比

维度说明
前端职责三方案卡片（A 丹麦海运 / B 中国空运 / C 当地陆运），各成本项+总成本；汇总对比表，最优方案绿色高亮 + ⭐ 推荐
后端职责POST /api/cases/{id}/plans：三次并行 AI 调用（asyncio.gather），结果写入 plan + plan_cost_items 表
依赖P0-4（以审核后成本为基准，方案在此基础上调整运输/关税/时间相关成本）

4.6 P1-6：规则管理

维度说明
前端职责规则列表表格 + 添加/编辑弹窗（规则名、适用客户下拉、规则类型、作用值）；开关启停
后端职责CRUD /api/rules；规则匹配引擎 — AI 推断时自动注入匹配规则作为 prompt 约束
依赖P0-3（规则在案例创建时生效）

4.7 P1-7：结果导出

维度说明
前端职责导出按钮，papaparse.unparse() + file-saver 生成 CSV 下载
后端职责可选：GET /api/cases/{id}/export 返回文件流
依赖P0-4 + P0-5

5.2 表结构清单

raw_cost_records — 原始成本记录

CREATE TABLE raw_cost_records (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    upload_batch_id TEXT    NOT NULL,           -- 上传批次 UUID
    platform        TEXT,    component         TEXT,
    site_name       TEXT    NOT NULL,           -- 用于 Country 映射
    cost_group      TEXT    NOT NULL,           -- Crane/Freight/Labour/Other/Parts/Tools
    cost_type_description TEXT, cost_per_unit  REAL,
    currency        TEXT,    exchange_rate_eur REAL,
    transfer_price_eur_net_incl_claim_sum REAL,
    quantity_calc_sum       REAL,
    failure_events          INTEGER,
    transfer_price_eur_sum  REAL,
    transfer_price_eur_claim_sum REAL,
    created_at      TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX idx_raw_cost_group ON raw_cost_records(cost_group);
CREATE INDEX idx_raw_site_name   ON raw_cost_records(site_name);

cost_group_mapping — CostGroup → 业务成本映射

CREATE TABLE cost_group_mapping (
    id                      INTEGER PRIMARY KEY AUTOINCREMENT,
    cost_group_value        TEXT NOT NULL UNIQUE,   -- 'Parts', 'Crane', etc.
    business_cost_category  TEXT NOT NULL,          -- '零部件成本', '吊车成本', etc.
    is_user_defined         INTEGER DEFAULT 0,      -- 0=系统默认, 1=用户修改
    updated_at              TEXT NOT NULL DEFAULT (datetime('now'))
);

cases — 案例主表

CREATE TABLE cases (
    id                  INTEGER PRIMARY KEY AUTOINCREMENT,
    case_name           TEXT NOT NULL,
    -- 🟡 用户输入区 (7 字段)
    turbine_model       TEXT NOT NULL,    country        TEXT NOT NULL,
    project_name        TEXT NOT NULL,    contract_type  TEXT NOT NULL,
    fault_description   TEXT NOT NULL,
    repair_duration_hours REAL NOT NULL,
    penalty_amount_eur  REAL NOT NULL DEFAULT 0,
    -- 状态
    status              TEXT NOT NULL DEFAULT 'draft',
    upload_batch_id     TEXT,
    created_at          TEXT NOT NULL DEFAULT (datetime('now')),
    updated_at          TEXT NOT NULL DEFAULT (datetime('now'))
);

case_cost_items — 案例成本项（每案例 8 行）

CREATE TABLE case_cost_items (
    id                      INTEGER PRIMARY KEY AUTOINCREMENT,
    case_id                 INTEGER NOT NULL REFERENCES cases(id) ON DELETE CASCADE,
    business_cost_category  TEXT NOT NULL,       -- 7 类 + '总成本合计'
    ai_inferred_value       REAL,    ai_confidence    REAL,
    ai_reasoning            TEXT,    reviewed_value   REAL,
    is_overridden           INTEGER DEFAULT 0,  override_reason  TEXT,
    source_record_count     INTEGER, source_avg_cost  REAL,
    created_at              TEXT NOT NULL DEFAULT (datetime('now')),
    updated_at              TEXT NOT NULL DEFAULT (datetime('now'))
);

plans + plan_cost_items — 方案与明细

CREATE TABLE plans (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    case_id         INTEGER NOT NULL REFERENCES cases(id) ON DELETE CASCADE,
    plan_type       TEXT NOT NULL,       -- 'denmark_sea'|'china_air'|'local_land'
    plan_label      TEXT NOT NULL,       -- '方案A：丹麦海运'|...
    total_cost_eur  REAL,   total_duration_days REAL,
    comparison_rank INTEGER,             -- 排名 1/2/3
    ai_reasoning    TEXT,
    created_at      TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE plan_cost_items (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    plan_id         INTEGER NOT NULL REFERENCES plans(id) ON DELETE CASCADE,
    business_cost_category TEXT NOT NULL,
    estimated_value REAL,   ai_reasoning   TEXT,
    created_at      TEXT NOT NULL DEFAULT (datetime('now'))
);

rules — 规则表

CREATE TABLE rules (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    name            TEXT NOT NULL,   description TEXT,

    rule_type       TEXT NOT NULL,   -- 'multiply'|'add'|'cap'|'floor'
    rule_value      REAL NOT NULL,   condition_json TEXT,
    priority        INTEGER DEFAULT 0, enabled INTEGER DEFAULT 1,
    created_at      TEXT NOT NULL DEFAULT (datetime('now')),
    updated_at      TEXT NOT NULL DEFAULT (datetime('now'))
);

7 AI 集成策略

7.1 异步通信核心代码

class AIInferenceService:
    def __init__(self, config: AIConfig):
        self.client = httpx.AsyncClient(
            base_url=config.ollama_base_url,
            timeout=httpx.Timeout(120.0)
        )
        self.model = config.model_name

    async def chat_stream(self, messages: list[dict]) -> AsyncIterator[str]:
        """流式调用 Ollama，逐 token yield"""
        async with self.client.stream(
            "POST", "/api/chat",
            json={
                "model": self.model,
                "messages": messages,
                "stream": True,
                "options": {
                    "temperature": 0.3,
                    "top_p": 0.9,
                    "num_predict": 4096
                }
            }
        ) as response:
            async for line in response.aiter_lines():
                if line.strip():
                    data = json.loads(line)
                    if data.get("done"): break
                    yield data["message"]["content"]

7.2 三套 Prompt 工程设计

Prompt ① — CostGroup 映射验证

角色内容
System你是风电运维成本分析专家。CostGroup 有 6 个值：Crane, Freight, Labour, Other, Parts, Tools。请理解它们对应的业务成本类型。
User给出 CostGroup 映射（含 CostTypeDescription 统计分布），验证默认映射是否合理。
输出JSON 格式的映射确认或修正建议。

Prompt ② — 案例成本推断（核心）

角色内容
System你是风电运维成本估算专家。根据历史成本数据和故障描述，推断 7 类成本。优先同 Component+同 Country 匹配，降级到全局平均。输出严格 JSON。
统计注入历史数据摘要：各 CostGroup 均值/中位数/样本数；各 Component 成本分布；各 Country 成本分布。
User案例 7 字段 + 匹配的规则列表。
输出{"items": [{business_cost_category, estimated_value, confidence, reasoning, source_record_count}], "total_estimated_cost": ...}

Prompt ③ — 三方案生成

角色内容
System你是风电运维物流方案专家。基于审核后的案例，生成 A（丹麦海运）、B（中国空运）、C（当地陆运）三方案。每方案调整运输/关税/时间相关成本。输出严格 JSON 含排名和理由。
User审核确认的完整案例数据 + 规则。
输出{"plans": [{plan_type, items[], total_cost_eur, total_duration_days, comparison_rank, reasoning}]}

7.3 错误处理策略

场景处理方式
Ollama 进程未启动前端提示"AI 服务不可用，请确认 Ollama 已启动" + 启动指令
模型未安装调用前检查 /api/tags，若模型不在则提示 ollama pull qwen2.5:14b
推理超时（120s）前端显示超时提示，允许重试；后端记录日志
JSON 解析失败自动重试最多 2 次，重试 prompt 中加入"请确保输出严格 JSON"
输出字段缺失用历史中位数填充，标记 confidence=0.1
笔记本 RAM 不足监控推理时内存；若 OOM，建议切换到 qwen2.5:7b
8 关键技术决策与权衡

8.1 决策记录（Top 10）

#决策✅ 选择❌ 放弃权衡分析
D1后端语言Python + FastAPINode.js + ExpressPython 是 AI/ML 母语；Pandas 处理 CSV 天然优势；AI prompt 构建涉及大量统计计算
D2AI 通信SSE 流式透传轮询 / WebSocketOllama 原生支持 stream，SSE 是 HTTP 原生方案，比 WebSocket 更简单
D3前端状态ZustandRedux ToolkitAPI 表面积最小，3 个 store 对 Demo 级项目足够
D4表格性能ProTable 分页虚拟滚动100 行/页分页即可满足需求，虚拟滚动仅在全量展示时才必要
D5成本推断单次 AI 调 7 类7 次独立调用单次调用保留成本间关联性，总延迟 1 次推理（vs 7 次串联）
D6Prompt 数据注入统计摘要全量 5,000 行统计摘要在 2-3K tokens 内传递数据分布，避免超 token 限制
D7方案生成三次并行调用一次调用三方案并行 = 总延迟≈最慢的一次；独立上下文保证每个方案质量
D8CSV 导出前端浏览器生成后端文件流papaparse 直接从 store 取数据生成 Blob，无需额外网络请求
D9Monorepo手动 workspaceNx / Turborepo2 个包不需要 task orchestration；concurrently 启动前后端即可
D10无日期字段等权处理按批次降权用户明确接受"无时间加权"；等权简化统计逻辑
